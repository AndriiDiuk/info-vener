Техническое задание на HTML верстку сайта
________________


1. Общие сведения о проекте
1.1. Тип проекта
1. Проект: сайт веб-студии Venner Digital.
2. Верстка: чистый HTML + CSS + минимальный JS.
3. CMS не используется. Верстка должна быть независимой от конкретного движка.
1.2. Исходные материалы от заказчика
1. Figma:
   * макеты для десктопа;
   * макеты для мобильной версии.
2. Google-таблица:
   * полный список URL страниц;
   * Title, Description, Keywords;
   * OG-title, OG-description, OG-url, OG-image;
   * при необходимости - дополнительные служебные поля.
3. Облако (Google Drive / Яндекс.Диск):
   * тексты для всех страниц;
   * юридические документы (политика, оферта, согласия и т.п.).
Исполнитель обязан использовать только эти материалы, а не придумывать тексты и метатеги самостоятельно.
1.3. Общий результат работы
1. Архив с полностью сверстанным сайтом, готовым к загрузке на сервер.
2. Проект должен:
   * соответствовать макетам;
   * корректно отображаться на основных разрешениях;
   * проходить проверку Google PageSpeed (Desktop и Mobile ≥ 95);
   * иметь валидный HTML-код и корректную микроразметку;
   * соответствовать требованиям Яндекса и Google по SEO.
________________


2. Структура проекта и include-логика
2.1. Структура папок
Обязательная структура:
1. /css - все стили.
2. /js - все скрипты.
3. /img - изображения. Подпапки по смыслу: /img/services, /img/blog, /img/icons и т.п.
4. /fonts - шрифты.
5. /include - все сквозные блоки, подключаемые через include.
6. /errors - страницы ошибок (400, 401, 403, 404, 500, 502, 503, 504).
Пути к файлам должны быть относительными и аккуратными, без пробелов и кириллицы.
2.2. Сквозные include-блоки
Через include реализуются:
1. header - логотип, меню, контакты, кнопки.
2. footer - копирайт, контакты, юридические ссылки, логотип.
3. Блок контактов:
   * телефон(ы);
   * email;
   * мессенджеры (WhatsApp, Telegram, Max и т.п.);
   * адрес / город.
4. Повторяющиеся CTA-кнопки:
   * «Рассчитать стоимость»;
   * «Заказать сайт»;
   * «Оставить заявку» и др.
5. Юридический блок (ссылки на политику, оферту, ПДн, cookies).
6. При необходимости - блок с логотипами соцсетей и мессенджеров.
Все эти данные должны правиться в одном месте, чтобы не дублировать текст по всему проекту.
2.3. Запреты по структуре
1. Нельзя копировать один и тот же HTML-блок на десяток страниц руками, если он повторяется - выносить в include.
2. Нельзя смешивать служебный код (метрика, квизы и т.п.) с версткой страниц - всё это подключается через Google Tag Manager, кроме базовых случаев, на которые укажет заказчик.

3. Семантическая верстка и AI-ready HTML
3.1. Семантические теги
Использовать:
1. <header> - шапка страницы или отдельного блока.
2. <main> - основной контент страницы (один на страницу).
3. <footer> - подвал.
4. <nav> - навигация (главное меню, вторичные меню).
5. <section> - крупные смысловые блоки.
6. <article> - самостоятельные материалы: статья, карточка блога, кейс.
7. <aside> - боковые блоки, дополнительные материалы.
Запрещено верстать всё подряд в <div> без логики.
3.2. Иерархия заголовков
1. На каждой странице должен быть один <h1> - заголовок страницы.
2. Далее используем <h2>, потом <h3> и т.д. без пропуска уровней.
3. Заголовки не используются только ради крупного шрифта - смысл важнее.
4. Текст заголовков сверяем с таблицей метатегов и макетом.
3.3. Статьи и раздел блога
Для каждой статьи:
1. Структура вверху:
   * <h1> - заголовок статьи;
   * блок краткого вывода (summary):
<section class="summary">
 <h2>Краткий вывод</h2>
 <p>Коротко, в 2–4 предложениях, о чем статья и к какому выводу она приводит.</p>
</section>
2. Основной текст далее разбивается на логические <section> с <h2>/<h3>.
3. Не допускается одна сплошная простыня текста без подзаголовков.
Это важно для:
* ИИ-выдачи Google (Search Generative Experience, AI-ответы);
* избранных сниппетов (Featured Snippets);
* удобства чтения.
3.4. Списки и таблицы
1. Списки всегда верстаются через <ul> или <ol>, а не через br.
2. Табличные данные всегда через <table>.
3. В таблицах использовать <thead>, <tbody>, <th>.
ИИ-агенты лучше понимают структурированные списки и таблицы, чем сырые <div>.
________________


4. Адаптив, mobile-first и доступность
4.1. Адаптивная сетка
1. Сайт должен корректно отображаться минимум на следующих ширинах: 320, 360, 414, 768, 1024, 1280, 1366, 1440, 1920.
2. Не допускается горизонтальный скролл.
3. Блоки не должны «ломаться» при переносе текста.
4.2. Mobile-first
1. Google оценивает сайт в первую очередь по мобильной версии.
2. Мобильная верстка не должна быть «обрезанной» - тот же контент, что и на десктопе (кроме второстепенных декоративных элементов).
Важно: нельзя прятать важный SEO-текст через display: none или убирать его только на мобильной версии.
4.3. Доступность (A11y)
1. Минимальный размер интерактивных элементов для тач-экрана - 44x44px.
2. Все важные кнопки и ссылки должны иметь фокус-состояние (:focus).
3. Контраст текста и фона должен быть достаточным для чтения (не бледный серый на белом).
4. Используем aria-label для:
   * иконок кнопок без текста;
   * элементов управления (бургер-меню, закрытие попапа).
5. Страница должна быть проходима клавишей Tab в логичном порядке.
________________


5. Изображения, медиа и Core Web Vitals
5.1. Форматы и размеры
1. Основной формат изображений - WebP.
2. Допускается PNG/JPEG в отдельных случаях (логотип, если нужна прозрачность, и т.п.).
3. Требования к весу:
   * крупные баннеры: до 300–500 КБ;
   * обычные изображения: до 200–300 КБ;
   * иконки: до 30–50 КБ.
5.2. Атрибуты изображений
1. Для всех <img> обязательно прописываем width и height - это важно для CLS.
2. Все контентные изображения имеют осмысленный alt.
3. Декоративные изображения - alt="".
5.3. Lazy load и responsive
1. Все изображения ниже первого экрана подключаются с ленивой загрузкой (loading="lazy" либо через JS-скрипт).
2. Для крупных баннеров желательно использовать srcset и sizes, если есть отдельные версии для мобильных/десктопа.
5.4. Core Web Vitals
Цели:
1. PageSpeed:
   * Desktop: ≥ 95;
   * Mobile: ≥ 95 для всех ключевых страниц.
2. LCP (Largest Contentful Paint) - до 2.5 секунд.
3. CLS (Cumulative Layout Shift) - ниже 0.1.
4. INP (Interaction to Next Paint) - до 200 мс.
Что важно для верстальщика:
1. Не блокировать поток тяжелыми синхронными скриптами.
2. Все JS-файлы подключать с defer или async, если возможно.
3. Критические стили первого экрана (critical CSS) - в <head>, остальное можно грузить ниже.
4. Не использовать тяжелые UI-фреймворки без необходимости (Bootstrap, большие CSS-библиотеки).
________________


6. SEO-основа: метатеги, Open Graph, robots, sitemap
6.1. Метатеги
На каждой странице в <head> обязательно:
1. <title> - подтягивается из Google-таблицы.
2. <meta name="description"> - также из таблицы.
3. <meta name="keywords"> - если используется.
4. <meta charset="utf-8">.
5. <meta name="viewport" content="width=device-width, initial-scale=1.0">.
Заголовок H1 должен логично соответствовать title, но не копировать его слово в слово.
6.2. Open Graph
Каждая страница сайта, без исключений (включая статьи, портфолио и т.п.), должна иметь:
1. <meta property="og:title" content="...">.
2. <meta property="og:description" content="...">.
3. <meta property="og:url" content="https://site.ru/адрес-страницы/">.
4. <meta property="og:image" content="https://site.ru/img/og/имя-файла.webp">.
5. <meta property="og:type" content="website"> или article для статей.
Требования к og:image:
1. Размер 1200×630 px.
2. Формат WebP или JPG.
3. Файл физически существует и доступен без авторизации.
4. URL абсолютный, с https.
6.3. Canonical и дубль-страницы
1. На всех страницах должен быть <link rel="canonical" ...>.
2. Следить за тем, чтобы:
   * не было дублей с / и без / на конце;
   * не было дублей http/https, www/без www.
6.4. Robots.txt и sitemap.xml
1. Верстальщик готовит базовые шаблоны этих файлов.
2. В sitemap не должны попадать:
   * страницы ошибок (4xx, 5xx);
   * технические страницы, не предназначенные для индексации (если заказчик отдельно это укажет).
________________


7. Микроразметка schema.org и интеграция с AI-поиском
7.1. JSON-LD
1. Используем только формат JSON-LD (скрипт <script type="application/ld+json"> в <head> или перед закрывающим </body>).
2. Структура JSON берется из подготовленных шаблонов заказчика.
3. Верстальщик подставляет:
   * название компании;
   * домен;
   * контакты;
   * адрес;
   * социальные сети;
   * URL конкретной страницы.
7.2. Типы схем
В зависимости от страницы используются:
1. Organization - данные компании (логотип, адрес, телефоны).
2. WebSite - общая информация о сайте.
3. WebPage - для каждой страницы.
4. Service - для страниц услуг.
5. FAQPage - для блока с вопросами-ответами.
6. BlogPosting - для статей.
7. ItemList / CollectionPage - для списков услуг, статей и т.п.
7.3. Speakable (опционально для статей)
1. Для блока summary в статье можно добавить разметку speakable.
2. Это облегчает работу голосовым ассистентам и ИИ-выдаче.
________________


8. Формы, скрипты и интеграции
8.1. Формы
1. Все формы должны иметь:
   * явные подписи к полям;
   * валидацию (минимум HTML5 - required, корректный тип поля для e-mail, телефона и т.п.);
   * маску телефонного номера, если это согласовано.
2. После успешной отправки должно быть понятное пользователю сообщение.
3. При ошибке - указать, что именно не заполнено или заполнено неверно.
8.2. Согласие с ПДн
1. Под формами должен быть чекбокс с текстом согласия.
2. В тексте указывается ссылка на соответствующий юридический документ (политика, условия).
3. По умолчанию чекбокс не активен.
8.3. Интеграции
1. Счетчики и сторонние сервисы подключаются через Google Tag Manager:
   * Яндекс.Метрика;
   * Marquiz;
   * SmartCaptcha;
   * виджеты отзывов и т.п.
2. Нельзя хаотично вставлять полные скрипты сервисов на каждую страницу.
8.4. JS и библиотеки
1. Использовать чистый JS там, где это возможно.
2. Библиотеки (jQuery, слайдеры) допускаются только по согласованию.
3. Запрещено использование громоздких UI-фреймворков, если задача решается простой версткой.
________________


9. Страницы ошибок и HTTP-коды
9.1. Набор страниц ошибок
В папке /errors должны лежать:
1. 400.html - неверный запрос.
2. 401.html - требуется авторизация.
3. 403.html - доступ запрещен.
4. 404.html - страница не найдена.
5. 500.html - внутренняя ошибка сервера.
6. 502.html - плохой шлюз.
7. 503.html - сервис недоступен.
8. 504.html - шлюз не отвечает.
Дизайн шаблонов предоставляется заказчиком или берется с основного сайта Venner Digital.
9.2. Требования к страницам ошибок
1. Все страницы ошибок должны:
   * иметь понятный текст для пользователя;
   * содержать кнопку перехода на главную;
   * иметь базовый набор метатегов и OG (как минимум для 404);
   * содержать ссылку/упоминание Venner Digital (микро-реклама).
2. С точки зрения сервера каждая страница должна отдавать корректный HTTP-код (настройка на стороне сервера, но верстка должна быть готова).
________________


10. Юридические страницы и ПДн
10.1. Обязательные страницы
Сайт должен содержать следующие страницы:
1. Политика конфиденциальности.
2. Согласие на обработку персональных данных.
3. Публичная оферта (если предусмотрена).
4. Условия соглашения.
5. Отказ от ответственности.
6. Страница о cookies / cookies solutions.
7. При необходимости - страница о капче (SmartCaptcha) и т.п.
Тексты предоставляет заказчик.
10.2. Размещение и доступность
1. Ссылки на все юридические страницы должны находиться в футере сайта.
2. Страницы доступны без авторизации, имеют чистые URL и корректные H1.
________________


11. Запрещено и типичные ошибки
11.1. Верстка
1. Нельзя использовать инлайновые стили, кроме редких согласованных случаев.
2. Нельзя делать большие фрагменты текста картинками.
3. Нельзя скрывать важный SEO-текст через display:none или уменьшением до 1px.
4. Нельзя использовать <br> для отступов и визуальной разметки.
11.2. JS
1. Нельзя блокировать интерфейс тяжелыми синхронными скриптами.
2. Нельзя использовать document.write и другие устаревшие подходы.
11.3. SEO
1. Нельзя оставлять страницы без Title и Description.
2. Нельзя дублировать Title и H1 на множестве разных страниц, если это не согласовано.
3. Нельзя допускать дублей URL с разными вариациями (www, http/https, слэш на конце и т.п.).
________________


12. Критерии приемки работы
Работа считается выполненной, если:
1. Все страницы соответствуют макетам Figma (по структуре, цветам, типографике).
2. Сайт корректно работает на основных разрешениях и в основных браузерах.
3. PageSpeed по ключевым страницам:
   * Desktop ≥ 95;
   * Mobile ≥ 95.
4. Core Web Vitals находятся в зеленой зоне (LCP, CLS, INP).
5. HTML-код валиден, нет критических ошибок в консоли.
6. Микроразметка JSON-LD присутствует и проходит базовую проверку в валидаторе.
7. На всех страницах есть:
   * Title, Description, H1;
   * OG-набор, включая og:image;
   * корректный canonical.
8. Работают все формы, есть валидация, чекбоксы согласия и ссылки на юридические документы.
9. Все страницы ошибок отдают правильные HTTP-коды и используют верстку из /errors.
10. Нет критичных визуальных багов, «прыгающих» элементов и горизонтального скролла.
